# Architecture Specification
**Проект:** Интерактивная платформа «TELL ME PLEASE»
**Тип архитектуры:** Браузерный гибрид (Сlient-STT / Server-LLM+TTS) [3]

## 1. Системный ландшафт (High-Level Design)
Система спроектирована для обеспечения минимальной задержки (TTFA < 1 сек) при одновременной нагрузке до 100 активных голосовых сессий [1, 2].

### 1.1. Frontend (Клиент)
* **Стек:** Next.js (App Router), TailwindCSS, Framer Motion, Vanilla JS для работы со звуком.
* **Распознавание речи (STT):** `Web Speech API`. Работает локально в браузере клиента, переводит речь в текст и отправляет на сервер готовые строки [4, 5]. Нулевая нагрузка на сервер, нулевая стоимость [5].
* **Воспроизведение (Audio Playback):** `AudioWorklet` используется для непрерывного воспроизведения потоковых аудио-чанков без задержек и артефактов.

### 1.2. Backend (Серверная оркестрация)
* **Стек:** Python 3.10+, FastAPI, WebSockets.
* **Роль:** Принимает текст от клиента, управляет запросами к LLM, буферизует предложения и проксирует аудио от TTS обратно клиенту [6, 7].
* **Менеджер ключей (Key Rotation Manager):** Кастомный модуль, хранящий пул API-ключей OpenCode Zen. Автоматически меняет ключ при ошибках лимита (HTTP 429/403) и делает retry запроса [8].
* **Маршрутизатор промптов (State Machine):** LLM работает без динамической генерации системных промптов. Бэкенд определяет ветку (класс) по ID, переданному с фронтенда, и подставляет жестко заданный (hardcoded) системный промпт [1, 9]. 

### 1.3. LLM Engine (Генерация текста)
* **Провайдер:** OpenCode Zen (`https://opencode.ai/zen/v1/chat/completions`) [10].
* **Модель:** `deepseek-v4-flash-free` [11].
* **Оптимизация:** Модель работает в режиме потоковой генерации токенов (`stream=True`). Бэкенд поддерживает "скользящее окно контекста", сохраняя только последние 10-12 реплик для экономии токенов и предотвращения ошибки переполнения 200К токенов [12, 13].

### 1.4. TTS Engine (Синтез речи)
* **Движок:** `Kokoro-82M` (через Docker-контейнер `Kokoro-FastAPI`) [14].
* **Интеграция:** Локальное развертывание на GPU-сервере бэкенда. Обеспечивает генерацию высококачественного аудио со скоростью до 100x от реального времени (RTF ~0.01) [15].

## 2. Поток данных (Data Flow / Streaming Pipeline)
Архитектура строго придерживается парадигмы потокового конвейера (Streaming Pipeline) с перекрытием фаз [2, 6]:

1. **Client STT:** Пользователь говорит $\rightarrow$ `Web Speech API` мгновенно транскрибирует фразу $\rightarrow$ отправляет текстовую строку по WebSocket на `FastAPI`.
2. **Backend LLM:** `FastAPI` выбирает ключ $\rightarrow$ делает запрос в `OpenCode Zen` $\rightarrow$ LLM начинает возвращать токены [7].
3. **Sentence Buffer:** `FastAPI` собирает токены. Как только обнаруживается знак препинания (., !, ?), сформированное предложение мгновенно отправляется в `Kokoro-FastAPI` [6, 7].
4. **Backend TTS:** `Kokoro-FastAPI` синтезирует аудио-чанк предложения $\rightarrow$ возвращает его в `FastAPI` [6].
5. **Client Playback:** `FastAPI` отправляет аудио-чанк по WebSocket на Frontend $\rightarrow$ `AudioWorklet` начинает воспроизведение.
*Пока клиент слушает первое предложение, LLM и TTS параллельно генерируют второе и третье.*

## 3. Управление сессией и прерывания
* **Защита от эха (Echo Feedback Loop):** В момент начала воспроизведения аудио клиентом, захват звука с микрофона программно приостанавливается (mute) во избежание попадания синтезированной речи обратно в STT. Запись возобновляется строго по событию окончания воспроизведения [13, 16].
* **Тайм-менеджмент:** На сервере и клиенте работает таймер сессии (3 минуты). По истечении времени WebSocket-соединение закрывается по инициативе сервера после отправки финального фидбека [17].
